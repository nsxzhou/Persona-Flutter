# Novel Workshop First-Batch Design Notes

This module is the Persona-Flutter home for long-form novel writing. The first
batch added executable context assembly and local design evidence. The second
batch adds minimal Drift persistence for runtime memory, chapter plans, and
single-record chapter content. It still does not add routes, UI, or live LLM
chapter generation.

## Current Persona-Flutter Evidence

| Area | Confirmed state |
| --- | --- |
| Project asset binding | `ProjectRecords` and `WritingProject` already expose `styleProfileId` and `plotProfileId`; the project dialog already lets users select both profiles. |
| Voice Profile asset | `StyleProfile.profileMarkdown` stores the reusable writing-style prompt asset generated by Style Lab. |
| Story Engine asset | `PlotProfile.storyEngineMarkdown` stores the reusable plot prompt asset; `plotSkeletonMarkdown` is available as the first source for automatic chapter planning. |
| Workflow trace | `WorkflowTaskRecords` and `WorkflowPromptTraceRecords` already support auditable LLM calls and should be reused for future chapter generation, continuity audit, and memory sync. |
| Persistence boundary | `schemaVersion` is currently 12. Version 11 removed legacy novel-workshop tables; version 12 reintroduces only the approved MVP tables. |

## Reference Capability Mapping

| Reference | Capability to absorb | Persona-Flutter mapping |
| --- | --- | --- |
| `Persona_副本` | Context assembly that layers reusable prompt assets, project context, and runtime state. | `WritingContextAssembler` assembles `Output Contract`, `Chapter Objective Card`, `Voice Profile`, `Story Engine`, `Project Context`, `Runtime Memory`, and `Writing Rules` in a fixed order. |
| `Persona_副本` | Memory Sync proposes updates after prose generation and writes back only after user confirmation. | Later batches store memory proposals on the chapter record and keep project-level runtime memory unchanged until accepted. |
| `Persona_副本` | Continuity review checks character drift, world-rule conflict, unfulfilled hooks, and chapter objective completion. | Later chapter workflow records audit verdict/report before saving or rejecting generated prose. |
| `Persona_副本` | Prompt layering keeps style, plot, project setting, and task layers from overstepping. | The assembler keeps each section separate and injects raw Voice Profile / Story Engine Markdown without rewriting them. |
| `tianming-novel-ai-writer` | Generation loop reads context, generates, gates, lands content, and writes state back. | Future workflow stages should be `assemble -> draft -> audit/gate -> save -> propose_memory_sync`. |
| `tianming-novel-ai-writer` | Fact snapshots structure characters, hooks, plot nodes, timelines, and world rules. | Fact snapshots remain a later derived index; Markdown runtime memory is the first source of truth. |
| `tianming-novel-ai-writer` | Generation gates validate protocol, references, consistency, unknown entities, and required appearances. | Later lightweight gates should produce `pass / warning / fail`; aesthetic judgment must not be a hard failure. |
| `tianming-novel-ai-writer` | Long-term direction includes long-range recall and whole-book validation. | These stay outside MVP until chapter workflow and human-confirmed memory sync are stable. |

## Non-Migration List

- Do not migrate WPF/.NET code or service architecture from `tianming-novel-ai-writer`.
- Do not restore the older FastAPI/Next.js runtime from `Persona_副本`.
- Do not introduce LangGraph or a new external agent state machine.
- Do not make the reference projects runtime dependencies.
- Do not reimplement Style Lab or Plot Lab generation inside Novel Workshop.

## Priority List

1. Reuse existing `Voice Profile` and `Story Engine` prompt assets.
2. Build deterministic context assembly and tests.
3. Use `ProjectRuntimeMemoryRecords`, `ChapterPlanRecords`, and `ProjectChapterRecords` for runtime memory and single-table chapter persistence.
4. Add generation, continuity audit, and memory-sync proposal flow through existing workflow traces.
5. Add derived fact snapshots and lightweight gates only after the writing loop is stable.

## Persistence Decisions

- Runtime memory is a single current row per project.
- Chapter plans are separate from chapter content so generated plans can be edited before prose generation.
- Chapter content uses a single row per chapter and is overwritten on edit/regeneration.
- When chapter content changes, prior memory-sync proposal fields are cleared and status returns to `idle`.
- A memory-sync proposal must match the current chapter content hash before it can be stored.
